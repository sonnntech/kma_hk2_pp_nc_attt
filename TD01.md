# QUY TRÌNH HOẠT ĐỘNG HỆ THỐNG BLOCKCHAIN DATA PIPELINE
*(Tài liệu báo cáo khoa học và sơ đồ luồng dữ liệu)*

Tài liệu này tổng hợp toàn bộ cấu trúc kiến trúc, luồng đi của dữ liệu (Data Flow) và quy trình kiểm toán toàn vẹn dữ liệu kết hợp sổ cái Blockchain giả lập cho dự án **blockchain-data-pipeline**. Các sơ đồ dưới đây sử dụng định dạng **Mermaid Diagram** được hỗ trợ hiển thị trực quan trực tiếp trên các trình duyệt Markdown hoặc các công cụ hỗ trợ (như VS Code, GitHub).

## 1. Sơ Đồ Kiến Trúc Hệ Thống (System Architecture Flowchart)

Sơ đồ này mô tả chi tiết từ lúc tạo dữ liệu giả lập, xử lý qua tác vụ ETL, băm dữ liệu chuẩn hóa, lưu trữ vật lý vào kho dữ liệu (Warehouse) và đồng bộ hóa bằng chứng lên chuỗi Blockchain.

```mermaid
flowchart TD
    %% Source & Agent Stage
    subgraph Source_Stage ["1. Nguồn Dữ Liệu & Đánh Giá An Ninh"]
        A[Mock Source Generator] -->|Sinh dữ liệu giao dịch thô| B[Raw Records]
        B -->|Đánh giá rủi ro nguồn| C[Security Agent]
        C -->|Đóng gói kết quả| D[Security Assessment]
    end

    %% ETL & Hashing Stage
    subgraph ETL_Stage ["2. ETL & Chuẩn Hóa Dữ Liệu"]
        B -->|Biến đổi & phân nhóm risk_band| E[Transformed Dataset]
        E -->|Chuẩn hóa JSON canonical_json| F[SHA256 Hashing]
        F -->|Tính toán| G[Dataset Hash]
        
        G & D -->|Kết hợp tạo Metadata| H[Lineage Manifest]
        H -->|Chuẩn hóa JSON canonical_json| I[Manifest Hash]
    end

    %% Storage & Commit Stage
    subgraph Storage_Stage ["3. Lưu Trữ Vật Lý & Ghi Bằng Chứng"]
        E & H -->|Ghi file JSON vật lý| J[(Data Warehouse: warehouse.json)]
        G & I -->|Lưu Proof| K[Contract Service]
    end

    %% Blockchain Ledger Stage
    subgraph Ledger_Stage ["4. Ký Số & Đào Block Ghi Blockchain"]
        K -->|Tạo payload| L[Transaction Service]
        L -->|Ký bằng Private Key| M[Signed Transaction]
        M -->|Mine block mới| N[(Deterministic Ledger: blockchain_ledger.json)]
        K -->|Đồng bộ hóa| O[(Contract State: contract_state.json)]
    end

    %% Audit Stage
    subgraph Audit_Stage ["5. Trình Kiểm Toán (Audit Engine)"]
        J -->|1. Đọc dữ liệu thực tế| P[Audit Engine]
        P -->|2. Tính lại Dataset Hash| Q[Re-calculated Hash]
        O -->|3. Lấy dataset_hash gốc| R[On-chain Proof]
        Q -->|4. So khớp| S{Trùng khớp?}
        S -->|Đúng| T[Verdict: Hash Match - Data Integrity Verified]
        S -->|Sai| U[Verdict: Hash Mismatch - Data Tampering Detected]
    end

    %% CSS Styling
    style J fill:#f9f,stroke:#333,stroke-width:2px
    style N fill:#bbf,stroke:#333,stroke-width:2px
    style O fill:#bbf,stroke:#333,stroke-width:2px
    style T fill:#8f8,stroke:#333,stroke-dasharray: 5 5
    style U fill:#f88,stroke:#333,stroke-dasharray: 5 5
```

## 2. Luồng Trình Tự Tương Tác Dữ Liệu (Data Flow Sequence Diagram)

Sơ đồ trình tự biểu diễn dòng thời gian tương tác giữa các lớp đối tượng trong mã nguồn Python từ đầu vào cho đến khi kết xuất kết quả kiểm toán:

```mermaid
sequenceDiagram
    autonumber
    participant Src as Mock Sources
    participant Sec as Security Agent
    participant ETL as ETL Pipeline
    participant WH as Warehouse
    participant CS as Contract Service
    participant Web3 as Web3 Ledger Client
    participant Audit as Audit Engine

    Note over Src, ETL: QUY TRÌNH CHẠY PIPELINE ETL & GHI PROOF
    Src->>ETL: generate_mock_source_data(count)
    ETL->>Sec: inspect_records(raw_records)
    Sec-->>ETL: SecurityAssessment (pass/review)
    ETL->>ETL: transform_records() [Thêm risk_band & amount_usd]
    ETL->>ETL: sha256_digest(transformed_records) -> dataset_hash
    ETL->>ETL: build_lineage_manifest()
    ETL->>WH: write_warehouse(warehouse.json, payload)
    ETL->>ETL: sha256_digest(manifest) -> manifest_hash

    Note over ETL, Web3: ĐỒNG BỘ HÓA LÊN BLOCKCHAIN
    ETL->>CS: store_proof(dataset_name, dataset_hash, manifest_hash)
    CS->>Web3: mine_transaction(tx_payload)
    Note over Web3: Tính toán Block Hash mới dạng:<br/>sha256(prev_hash : tx_hash : block_no)
    Web3-->>CS: Transaction Receipt (Mined)
    CS-->>ETL: Receipt & Proof Committed

    Note over WH, Audit: QUY TRÌNH KIỂM TOÁN TỰ ĐỘNG
    Audit->>WH: read_warehouse(warehouse.json)
    WH-->>Audit: records & manifest
    Audit->>Audit: sha256_digest(records) -> actual_hash
    Audit->>CS: get_proof(dataset_name)
    CS-->>Audit: Stored Proof (expected_hash, manifest_hash)
    Audit->>Audit: So sánh (actual_hash == expected_hash)
    Audit-->>ETL: VerificationResult (pass/fail)
```

## 3. Vòng Đời Trạng Thái & Phát Hiện Sửa Đổi Trái Phép (Lifecycle & Tamper Detection)

Sơ đồ mô tả quy trình giả lập tấn công sửa đổi cơ sở dữ liệu vật lý (tập tin `warehouse.json`) để chứng minh cơ chế phát hiện sai lệch của hệ thống:

```mermaid
stateDiagram-v2
    [*] --> InitialState: Khởi động hệ thống
    InitialState --> SmartContractDeployed: deploy.py (Tạo Artifact & Địa chỉ Contract)
    
    state PipelineExecution {
        SmartContractDeployed --> DataGenerated: generate_mock_source_data()
        DataGenerated --> DataTransformed: transform_records()
        DataTransformed --> ProofsCalculated: Tính hash cho Dataset & Manifest
        ProofsCalculated --> ProofCommitted: store_proof() (Ghi chép bất biến lên Ledger)
    }

    state AuditVerification {
        ProofCommitted --> IntegrityVerified: Chạy Audit Engine (Dữ liệu gốc khớp hoàn toàn)
        note right of IntegrityVerified
            Kết quả: "Hash Match - Data Integrity Verified"
        end note
        
        IntegrityVerified --> DataTampered: KỊCH BẢN TẤN CÔNG (simulate_tamper_attack.py)<br/>Sửa đổi file warehouse.json một cách trái phép
        DataTampered --> TamperDetected: Chạy Audit Engine (So sánh Hash mới với Proof on-chain)
        note right of TamperDetected
            Kết quả: "Hash Mismatch - Data Tampering Detected"
        end note
    }

    TamperDetected --> [*]: Hoàn thành quy trình minh họa PoC
```

## 4. Giải Thích Các Bước Chi Tiết Trong Quy Trình

### Bước 1: Thu thập và Đánh giá An ninh Nguồn (Mock Source & Security Assessment)
* **Thành phần**: `pipeline/mock_sources.py`, `pipeline/security_agent.py`
* **Mô tả**: Dữ liệu thô giả lập được tạo ra tự động. Trước khi đưa vào xử lý ETL, `SecurityAgent` sẽ quét tìm các bất thường chất lượng dữ liệu (ví dụ: giao dịch có số tiền âm hoặc bằng không). Trạng thái này (`pass` hoặc `review`) sẽ được đóng gói lại thành `SecurityAssessment`.

#### Quy Trình Sinh Dữ Liệu Thô (Raw Data Generation Flow)

Dữ liệu đầu vào của hệ thống được giả lập tự động nhằm kiểm thử đường ống ETL và cơ chế ký số. Tiến trình chạy sẽ lặp qua số lượng bản ghi được yêu cầu (`record_count`) và tính toán động các giá trị dựa trên chỉ mục (loop index):

```mermaid
flowchart TD
    Start([Bắt đầu sinh dữ liệu]) --> Loop{Lặp từ index = 0 đến count - 1}
    
    %% Quy trình tính toán động
    Loop -->|record_id| CalcID[record_id = index + 1]
    Loop -->|customer_id| CalcCust["customer_id = CUST- + (index+1 dạng 03d)"]
    Loop -->|amount| CalcAmt[amount = 125.0 + index * 17.5]
    Loop -->|currency| CalcCur[currency = USD]
    Loop -->|event_time| CalcTime[event_time = 2026-01-01T00:00:00Z + index * 1 giờ]
    Loop -->|region| CalcReg[region = Xoay vòng NA -> EU -> APAC]
    
    %% Tạo bản ghi
    CalcID & CalcCust & CalcAmt & CalcCur & CalcTime & CalcReg --> BuildRec[Đóng gói vào dataclass SourceRecord]
    BuildRec --> ConvertDict[Chuyển đổi thành Dictionary: asdict]
    ConvertDict --> AppendList[Thêm vào danh sách records]
    
    AppendList --> Next{Đạt số lượng?}
    Next -->|Chưa đủ| Loop
    Next -->|Đã đủ| End([Trả về danh sách dữ liệu thô dạng JSON List])
```

### Bước 2: Chuyển đổi ETL & Chuẩn Hóa Dữ Liệu (ETL & Canonical Hashing)
* **Thành phần**: `pipeline/etl_pipeline.py`, `pipeline/hashing.py`
* **Mô tả**: Dữ liệu được tính toán thêm cột `amount_usd` và xếp loại `risk_band` (High/Standard). Nhằm đảm bảo dữ liệu băm không bị ảnh hưởng bởi định dạng khoảng trắng hay thứ tự khóa trong tệp JSON, hệ thống sử dụng cơ chế **Canonical Serialization** trước khi đưa qua thuật toán băm SHA256 thu về `dataset_hash` duy nhất.

### Bước 3: Tạo Lineage Manifest & Lưu Dữ Liệu Vào Warehouse
* **Thành phần**: `pipeline/lineage.py`, `pipeline/warehouse_writer.py`
* **Mô tả**: Siêu dữ liệu truy vết nguồn gốc (gồm `dataset_hash`, thời gian chạy, số lượng bản ghi, ID tiến trình chạy pipeline và thông tin đánh giá an ninh) được đóng gói thành `LineageManifest`. Bản kê này sau đó được băm ra `manifest_hash`. Toàn bộ dữ liệu thực tế và manifest được ghi xuống tệp `warehouse/warehouse.json`.

### Bước 4: Cam Kết Bằng Chứng Bất Biến Lên Blockchain
* **Thành phần**: `blockchain/`, `contracts/DataPipelineGovernance.sol`
* **Mô tả**: Hai chữ ký số (`dataset_hash` và `manifest_hash`) được gửi lên smart contract giả lập thông qua hàm `storeProof`. Hệ thống giả lập các khối (blocks) tuần tự của blockchain bằng sổ cái `state/blockchain_ledger.json` được ký bằng khóa riêng mật mã học.

### Bước 5: Kiểm Toán Độc Lập Phát Hiện Thay Đổi
* **Thành phần**: `audit/audit_engine.py`
* **Mô tả**: Để xác thực tính toàn vẹn, Audit Engine đọc lại tệp `warehouse/warehouse.json`, tính toán lại hash và truy tìm bằng chứng gốc đã lưu trên blockchain thông qua hàm `getProof`. Nếu bất kỳ ai chỉnh sửa dù chỉ một ký tự hoặc số tiền của bất kỳ giao dịch nào trong warehouse, kết quả băm thực tế sẽ thay đổi, hệ thống sẽ ngay lập tức so khớp lệch và báo động hành vi phá hoại dữ liệu.
